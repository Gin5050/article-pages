どんな論文か

Shepherd: A Runtime Substrate Empowering Meta-Agents with a Formalized Execution Trace は、エージェントをただ実行するのではなく、その実行をあとから操作できる対象として扱うための runtime substrate を提案する論文。

普通の agent 実行では、モデルが考え、ツールを呼び、ファイルや環境を変更し、ログが残る。失敗したら、そのログを人間や別 agent が眺めて「次はこうしよう」と考える。しかし Shepherd は一歩進めて、実行途中の状態を fork し、別の分岐を走らせ、必要なら元に戻し、過去の地点から replay することを runtime の基本操作にする。

重要なのは、これは単なるログ管理ではないこと。論文は、agent の行動を typed event として Git-like な execution trace に記録し、agent process と filesystem を一緒に fork / replay できるようにする。つまり、会話履歴だけでなく、環境の状態まで含めて「その時点の agent」を再利用可能にする。

ざっくり言うと、Shepherd は「agent の実行を Git の branch のように扱えるようにする」論文。meta-agent はその branch を見て、介入し、分岐させ、別の経路を試せる。

この論文が言っている範囲

読む時に一番混乱しやすいのは、Shepherd が「過去経験検索システム」なのか「戻れる過去を作る実行基盤」なのか、という点。結論から言うと、Shepherd 本体は後者に近い。

Shepherd は、たくさんある過去の軌跡から今のタスクに似たものを自動検索する memory system ではない。論文の中心は、agent と environment の過去状態を正確に fork / replay できるようにし、その上で meta-agent が監督・反実仮想検証・訓練用分岐をできるようにすること。

Shepherd が作るもの

過去時点に戻って再開できる execution trace。agent の会話だけでなく、filesystem、scope、tool effect も含む。

Shepherd が直接は解かないもの

大量の過去軌跡から、今のタスクに合う経験を検索・選択する問題。ここは別の memory / retrieval 層が必要になる。

実験で示すもの

戻れる過去があると、live supervision、counterfactual replay、Tree-RL training という3種類の meta-agent が作れること。

したがって、この論文は「過去から何を選ぶべきか」の完全解ではなく、「選んだ過去を本当に実行可能な形で再利用するには、runtime がこうなっている必要がある」という土台の論文として読むとよい。

課題と貢献

問題設定

meta-agent が worker agent を改善・監督・学習に使おうとしても、現在の実行基盤では過去状態への正確な復帰や分岐が難しい。

中心アイデア

agent の task、effect、scope、execution trace を第一級の対象にし、実行履歴を typed event の永続構造として扱う。

貢献

実行基盤、Lean で mechanize された中核 semantics、Python framework、そして supervision / optimization / RL training の3用途を示す。

この論文の貢献は「新しい agent が賢い」ではなく、「agent を賢く使うための実行面をどう作るか」にある。おい丸が普段言っている harness engineering にかなり近い。

しくみ

4つの基本単位

Shepherd は functional programming の発想を agentic execution に持ち込み、effectful な処理を「何をするか」と「どう世界に反映するか」に分ける。論文では、agent 実行を扱うために4つの単位を第一級にする。

Task

agent の振る舞いそのもの。単なるプロンプトではなく、実行可能な関数として扱われる。

Effect

agent の行動。意図と結果を分けて記録し、可逆なもの、補償可能なもの、不可逆なものを区別する。

Scope

agent が走る環境。sandbox、model provider、tool surface を含み、fork / merge / discard の対象になる。

Execution trace

過去の状態を Git-like な commit graph として残す。過去地点へ戻り、そこから別の分岐を走らせられる。

fork / replay が何を変えるか

失敗した agent 実行を改善したい時、普通は最初から再実行するか、ログを読んでプロンプトやコードを直す。Shepherd では、失敗直前の地点を fork して、別の一手だけを試すことができる。これは debugging、supervision、RL rollout のいずれにも効く。

論文で面白いのは、filesystem だけでなく agent process と message prefix も一緒に扱うこと。これにより replay 時に prompt cache を再利用しやすく、meta-agent が多数の分岐を試す時のコストを抑えられる。

検証結果

論文は Shepherd を3つの使い方で評価する。どれも「実行を分岐できる」ことが直接効く設計になっている。ただし、ここで評価されているのは「過去軌跡検索」ではなく、「戻れる過去を持つ runtime substrate を使うと何ができるか」。

Live supervision

CooperBench の pair coding で、live supervisor による介入により pass rate が 28.8% から 54.7% へ上がる。

Counterfactual optimization

meta-agent が過去地点から別経路を探索し、4 benchmarks で MetaHarness や GEPA を上回る。最大 11 points 改善し、wall-clock も最大 58% 下がる。

Tree-RL training

meta-agent が選んだ turn から rollout を分岐し、TerminalBench-2 で Qwen3.5-35B-A3B の性能を 34.2% から 39.4% に上げる。

実験1: Live supervision

CooperBench で、2体の Claude Haiku 4.5 worker agent が並列に別々の feature を実装する。通常の並列実行では、互いの作業をうまく調整できず、片方の変更がもう片方を壊すような coordination failure が起きる。

Shepherd では、meta-agent が両 worker の effect stream を購読し、必要に応じて inject で助言を入れ、handoff で片方の scope をもう片方の新しい root にし、discard で詰まった worker を捨てる。ここでは「過去に戻る」より、実行中の状態を観測し、分岐・引き継ぎできることが効いている。

実験2: Counterfactual Replay Optimization

CRO は、agentic workflow の失敗 trace を meta-agent が読み、「この workflow edit なら失敗を直せるのでは」という候補を作る。各 edit は、直したい training examples の fix set と、悪化させたくない guard set を持つ。

Shepherd は parent trace のうち、その edit が影響し始める最初の event まで戻り、そこから suffix だけを replay する。全部を最初から走らせるのではなく、変更の影響を受ける下流だけを再実行するので、差分検証が速く、親 workflow との差も見やすい。候補が fix / guard の targeted gate を通れば dev set に回され、candidate pool に追加される。

実験3: Meta-Agent Guided Tree-RL

長期タスクでは、最後に成功か失敗かの reward だけが返ってくるため、途中のどの行動が効いたのか分かりにくい。Tree-GRPO では、meta-agent が root rollout の途中 turn を選び、Shepherd でその状態を fork して sibling branches を複数走らせる。

同じ prefix から分かれた未来の成否差を見ることで、suffix 側により細かい credit assignment を作る。ここでは「戻れる過去」は、過去経験の検索ではなく、訓練中の途中状態から複数の未来を試すために使われている。

結果の読みどころは、単に benchmark score が上がったことではない。fork/replay 可能な実行基盤があると、監督、探索、訓練が同じ substrate の上に乗る、という点が強い。

メタエージェントとは何か

この論文でいう meta-agent は、普通のタスクを直接解く agent ではなく、別の agent の実行を見て、指示したり、分岐させたり、改善したりする上位 agent のこと。worker agent が「コードを書く」「問題を解く」「ツールを呼ぶ」なら、meta-agent はその実行を観測・介入・評価・再実行する。

監督者

並列 worker の作業を見て、衝突しそうなら助言を入れたり、片方の scope をもう片方へ引き継がせる。

改善者

過去の失敗 trace を読んで workflow edit を提案し、counterfactual replay で本当に直るか検証する。

分岐点の選択者

訓練中の rollout のどこから branch すれば学習信号が取れそうかを選び、複数の未来を走らせる。

つまり、Shepherd における meta-agent は「賢い別モデル」というより、agent の実行履歴を操作対象にする agent。Shepherd は、その meta-agent に戻る・分ける・再開するための手足を渡している。

なぜ今読むか

直前に読んだ Direct Corpus Interaction は、agentic search において検索インターフェースを太くする話だった。Shepherd はそれに対して、agentic execution のインターフェースを太くする話に見える。

memory の議論では「何を保存するか」に目が行きがちだが、Shepherd を読むと「保存されるべき経験とは、そもそもどの粒度の実行なのか」という問いが出てくる。会話ログか、ツール呼び出しか、filesystem 差分か、sandbox 状態か、分岐した別経路か。ここを設計しないまま memory を足しても、再利用できる経験にはなりにくい。

限界と読みどころ

Shepherd は proof-of-existence 色が強い。つまり、これがそのまま全 agent platform の最終形というより、meta-agent が execution trace を操作できると何が可能になるかを示す論文として読むのがよい。

  • 不可逆な effect、たとえば外部送信や支払いなどは rollback できず、監査対象として記録するしかない。
  • counterfactual replay は、過去の編集と副作用が弱く結合していることをある程度仮定する。
  • meta-agent 自体のコストや品質が結果に影響するため、substrate と supervisor intelligence の寄与を分けて読む必要がある。
  • agent がこの substrate を自分で使いこなすには、post-training や UI / API 設計がさらに必要になる。

それでも、agent を「一回走らせてログを読むもの」から「分岐可能な実行 artifact を残すもの」へ移す補助線としてかなり強い。memory、skills、evaluation をまとめる下層として読める。

読後Q&A

Q. この論文の一番大きい主張は？

A. meta-agent が worker agent を監督・改善・訓練するには、実行履歴を分岐・巻き戻し・再生できる substrate が必要だ、という主張。

Q. 過去のある時点に戻って再開するの？

A. そう。agent と environment の過去状態を fork し、そこから別の未来を走らせる。ただし、全ての外部副作用が巻き戻せるわけではない。

Q. たくさんある過去軌跡から、今のタスクに合うものを選ぶの？

A. そこは Shepherd 本体の主張ではない。Shepherd は「戻れる過去」を作る。どの過去を検索・選択するかは、別の memory / retrieval / meta-agent 設計の問題。

Q. では実験では何をしたの？

A. 3用途で試している。実行中に監督する、workflow edit の影響地点から反実仮想 replay する、訓練中に途中 turn から sibling rollout を分岐する。

Q. meta-agent って何？

A. worker agent の実行を見て、介入・改善・分岐・評価する上位 agent。タスクを直接解くというより、タスクを解いている agent の実行を操作する。

Q. memory 論文なの？

A. 直接の memory 論文ではない。ただし「経験をどの粒度で artifact 化するか」という意味では、agent memory の前提を作る論文として読める。

Q. Git-like trace とは何？

A. agent の effect stream を commit graph のように扱うこと。過去状態を参照し、fork し、別経路を走らせられる。

Q. ただの checkpoint / restore と違う？

A. filesystem だけでなく、agent process、message prefix、tool effect、scope 操作を一体で扱うところが違う。

Q. 結果は何が強かった？

A. live supervision、counterfactual optimization、Tree-RL training の3つで、fork/replay できることが直接 score や wall-clock に効いている。

Q. 自分の運用にどう効く？

A. Discord / wiki / public page のログを残すだけでなく、agent 実行そのものを再利用可能な artifact として残す設計に目が向く。

Q. skill とどうつながる？

A. skill を改善するには、成功・失敗した実行のどの地点を変えればよかったかを見る必要がある。Shepherd はその分岐実験をしやすくする。

Q. harness engineering 的には？

A. harness は「agent の外側の配線」だけでなく、実行履歴、環境状態、分岐実験、評価を支える runtime でもある、という見方が得られる。

Q. 読後に残る一言は？

A. agent の経験は、ログとして保存するだけでは弱い。分岐して試せる execution trace にして初めて、改善や学習の材料になる。
