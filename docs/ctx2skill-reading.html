いま面白い理由

HTML artifact、Obsidian、Markdown、トークン効率の話は、見た目のフォーマット問題に見えます。 でも Ctx2Skill が扱うのは、その一段奥にある「文脈をどう再利用可能な作業知に変えるか」です。

context engineering agent skills memory wiki / AGENTS.md

長い文書を毎回プロンプトに入れるのは重い。要約だけだと、手順や判断基準が落ちる。 そこでこの論文は、文脈から「次のタスクでも使える自然言語 skill」を自動で取り出す。 ここが、ただの RAG や要約と違うところです。

中核のループ

Ctx2Skill の気持ちよさは、解く側だけでなく「良い問いを作る側」にも skill があるところです。

Challenger 文脈理解を試す課題と採点基準を作る。簡単に解かれたら、次はもっと鋭い問いを作るように更新される。
Reasoner 文脈と現在の skill を使って回答する。失敗したら、足りなかった文脈知識を skill に反映する。
Judge 採点基準ごとに pass / fail を返す。詳しい答えを教える先生ではなく、失敗を検知する圧力として働く。
ポイント: 更新されるのはモデル重みではなく、自然言語の skill set。だから閉じたモデルにも差し込めるし、wiki や AGENTS.md の運用にも近い。

Cross-Time Replay が効く理由

自己改善は、進めるほど良くなるとは限りません。後半の Challenger が極端な課題を作ると、Reasoner skill はその特殊ケースに寄りすぎます。 Cross-Time Replay は、途中で作られた複数の skill を代表課題で評価し直し、難しい課題にも簡単な課題にも崩れにくいものを選びます。

実務に置き換えると？

skill や AGENTS.md は、最後に直した版が常に最良とは限りません。直近の失敗に合わせすぎると、普通の依頼で過剰反応することがあります。

なぜ「代表課題」が必要？

改善対象の失敗だけで再評価すると、その失敗には強くなっても汎用性が落ちます。普段の成功ケースも含めて戻りテストする発想が大事です。

結果の読み方

数字は派手ではありません。でも、文脈学習が難しいタスクで一貫して改善している点を見るのがよさそうです。

GPT-4.1 11.1% → 16.5%
GPT-5.1 21.2% → 25.8%
評価カテゴリ 4 種類

絶対性能はまだ低いので、「これで文脈学習は解決」とは読まないほうがいいです。 ただ、skill を外部化して推論時に差し込む方向が、複数モデルで効いているのはかなりおいしい示唆です。

注意して読むところ

コスト: 各文脈ごとに課題生成、回答、採点、skill 更新を回すので、一回しか使わない文脈には重いです。

Judge 依存: 採点器が文脈理解の深い失敗を拾えるかは別問題です。pass / fail が正しければ強いけど、そこが揺れると危ない。

過剰適応: Cross-Time Replay があるとはいえ、自己対戦は特殊な課題へ寄りやすい。ここは実運用でも同じ匂いがします。
