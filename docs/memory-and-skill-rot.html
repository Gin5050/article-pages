Memory And Skill Rot In Agent Harnesses

long-running agent の memory と skill は、保存ファイルではなく、読み込み・適用・退役・統合・risk review を含む harness interface として設計する必要がある。増やすほど強くなるのではなく、古さ、重複、危険な副作用を管理できないと rot が起きる。

このメモの中心

このページは単一論文の要約ではなく、agent memory、agent skill、harness engineering をつなぐ公開用の読書 artifact である。中心にある問いは、長く動く agent が『覚える』だけで本当に良くなるのか、というもの。

結論はかなりはっきりしている。memory も skill も、保存された Markdown、DB row、SKILL.md だけでは完結しない。いつロードされるか、どんな根拠形式で提示されるか、どの条件で呼ばれるか、古いものをどう退役させるかまで含めて harness interface になる。

この見方に立つと、agent の長期運用で怖いのは不足よりも腐敗である。古い記憶、似た skill、危険な side effect、曖昧な呼び出し条件が増えると、保存量は増えているのに判断は不安定になる。

特に自然言語 skill は扱いやすい一方で、invocation interface、execution flow、tool/resource side effects、権限、失敗時の扱いが一つの文章に混ざりやすい。registry が大きくなるほど、discovery、risk review、重複統合が難しくなる。

だから memory と skill の設計は、追加する仕組みだけでなく、整理する仕組みとして考える必要がある。catalog、metadata、lint、usage history、archive、restore は地味だが、agent を長く使うほど中核部品になる。

このメモは、Agent Memory Engineering、From Skill Text to Skill Structure、skill lifecycle 管理の議論を、個人用 agent harness の設計論として束ねる。

memory transfer は file copy ではない。同じ情報を別の agent に渡しても、常時ロード、on-demand search、引用形式、system reminder、更新 API、削除や拒否の制約が違えば、使われ方は変わる。

skill も同じで、SKILL.md の本文だけを増やしても運用は強くならない。どの状況で呼ぶべきか、どの resource に触るか、どんな side effect を持つか、古くなったときにどう扱うかが必要になる。

拾っておきたい補助線

Memory is interface：

記憶の実体はファイルや行ではなく、読み込みタイミング、提示形式、更新・拒否・削除の動線まで含む harness interface である。

Skills need structure：

人間にも LLM にも読める skill 本文は便利だが、呼び出し条件、手順、副作用、リスクを機械可読に分けないと registry が育った時に崩れやすい。

Rot is lifecycle：

古い memory や skill を消せば終わりではない。使用頻度、最終利用時刻、patch 履歴、重複、drift、pin、archive、restore を持つ管理が必要になる。

Curation is capability：

catalog、lint、metadata、related link、status 更新は事務作業ではなく、agent の長期性能を守るための harness capability と見られる。

Harness として見る

1

まず memory を storage ではなく loading policy として見る。何を常時読むのか、何を必要時だけ探すのか、何を現在の判断基準として扱うのかを分ける。

2

次に presentation を見る。記憶を事実、方針、履歴、候補、警告のどれとして提示するかで agent の行動は変わる。根拠形式そのものが interface になる。

3

skill は Scheduling、Structural、Logical の三層で見る。いつ呼ぶか、どんな流れで動くか、どの atomic action と resource use を持つかを分ける。

4

最後に lifecycle を置く。active、draft、stale、archived、pinned のような状態と、usage count、last used、review history を使い、増やすだけの運用から抜ける。

運用で起きる失敗

Stale memory：

当時は正しかった記憶が今の状態や方針と合わなくなり、更新日や適用条件がないと古い前提が現在の根拠として混ざる。

Duplicate skills：

似た skill が複数あり、どれを優先するか分からないため、routing が揺れて似た作業なのに毎回違う手順へ入る。

Hidden side effects：

skill 本文に tool 操作、外部送信、ファイル更新、権限変更が自然文で埋もれ、risk review が呼び出し前に効きにくくなる。

Registry blindness：

skill や memory が増えすぎると存在を見つけられないか、見つけても古さや信頼度を判断できなくなる。

Accumulation trap：

learn-from-logs のような仕組みで知識を増やすほど賢くなるように見えても、統合と退役がなければ矛盾とノイズも増える。

注意点と読みどころ

このページは公開用の設計メモであり、単一実験の結果を報告する論文要約ではない。数値や評価結果は、参照元の論文や記事側で確認する必要がある。
自然言語 skill を否定しているわけではない。むしろ、人間が読める本文を残したうえで、機械可読な index、risk surface、lifecycle metadata を重ねるのが現実的な方向である。
memory や skill の退役は、単純な削除とは違う。古いものが再利用される場合、監査や復元のために archive と理由を残す設計が必要になる。
個人用 assistant と組織用 agent では、必要な governance の重さが違う。小さく始めるなら、updated、status、related、risk label くらいからでよい。

実務へ引くなら

  • wiki の catalog は、単なるファイル一覧ではなく retrieval policy の入口として扱う。summary、status、updated、related は agent が読む順序を決める信号になる。
  • skill metadata には、trigger、必要入力、触る resource、外部副作用、必要権限、last reviewed、stale 条件を持たせる余地がある。
  • wiki-lint は整合性確認だけでなく、memory rot を見つける curator として育てられる。薄い index、古い catalog、重複ページ、壊れた related link は performance issue でもある。
  • learn-from-logs は memory を増やす skill だが、将来的には memory を減らす、統合する、退役させる skill と対にした方がよい。
  • 公開ページや読書 artifact も同じで、古い手書き HTML と新しい JSON spec 生成ページが混在すると、更新フロー自体が rot する。生成元を持つことが保守性になる。

次に読むなら

  • From Skill Text to Skill Structure は、自然言語 skill を構造化するための三層表現を読む入口になる。
  • STALE は、古い記憶が現在の前提として残る問題を評価するための補助線になる。
  • Dynamic Skill Lifecycle Management は、skill を追加だけでなく retain、retire、expand の lifecycle として扱う視点をくれる。
  • Counterfactual Trace Auditing of LLM Agent Skills は、skill が成功率に出ない形で agent の行動を変えることを trace から監査する読み物としてつながる。

読後Q&A

このページの中心問いは？

long-running agent の memory と skill は、保存すればするほど良くなるのか、それとも古さや重複やリスクを管理しないと腐るのか、という問いである。

memory transfer が file copy ではないとは？

同じ内容を移しても、常時ロード、検索、引用、更新、拒否、削除の interface が違えば、agent の使い方も変わるという意味である。

skill rot とは何が腐ること？

手順そのものだけでなく、呼び出し条件、副作用、権限、前提となる tool や API、似た skill との優先順位が古くなり、registry 全体が扱いにくくなること。

自然言語 skill はだめなの？

だめではない。人間と LLM の両方が読める強みがある。ただし大きな registry では、trigger、resource use、risk、lifecycle state を別の metadata として持つ方が安全になる。

Scheduling、Structural、Logical の三層は何を見る？

Scheduling はいつ呼ぶか、Structural はどんな手順で動くか、Logical はどの atomic action と resource use を持つかを見る。

memory rot で一番怖いのは？

古い記憶が古いと分からないまま、現在の判断基準として使われ続けること。単に検索に出るだけならまだよいが、回答を支配すると危ない。

catalog や lint はなぜ重要？

agent にとっては、ファイルの存在より、どの情報が active で、何に関係し、いつ更新され、どれを優先して読むべきかが重要だから。catalog と lint はその信号を保つ。

まず何から導入すればいい？

小さく始めるなら、memory と skill に updated、status、related、risk label、last reviewed を持たせ、古いものを見つける lint を回すところからでよい。

この公開ページ自体を作り直す意味は？

古い手書き HTML では更新元が曖昧で、今の article-pages の生成フローと合わない。JSON spec を source of truth にすると、他ページと同じ形で保守できる。

一言でいうと？

agent を長く使うなら、覚える仕組みだけでなく、古さを見つけ、統合し、退役させる仕組みが必要である。
