Memory And Skill Rot In Agent Harnesses

long-running agent で増えていく memory と skill を、保存ファイルではなく、読み込み・適用・忘却・統合・risk review を含む harness interface として捉え直すための公開メモ。

中心の主張

long-running agent の問題は、記憶や skill を保存できるかではない。保存されたものが、いつ読まれ、どう提示され、どの条件で使われ、古くなった時にどう扱われるかが本体になる。

memory も skill も単なるファイル群ではなく、model と harness が合体する interface として設計する必要がある。保存、検索、適用、拒否、統合、廃棄までが一つの lifecycle になる。

雑に言うと、「たくさん覚えた agent」は強いとは限らない。古い判断基準、重複した手順、危険な記憶、呼び出し条件が曖昧な skill が増えると、能力ではなく負債が増える。

Memory は harness interface

memory transfer は file copy ではない。Claude Code、Codex、Hermes のような agent が似た内容を読めても、モデルはそれぞれ異なる memory UI、引用形式、読み込みタイミング、system reminder、更新 API に慣らされている。

つまり memory の実体は Markdown や DB row だけではない。常時ロードされるもの、on-demand search されるもの、根拠として提示される形式、古い記憶の扱い、危険な記憶を拒否するルールまで含めて memory layer になる。

Loading

何を最初から読むのか。何を検索時だけ読むのか。context の節約だけでなく、判断の初期条件を決める。

Presentation

記憶を根拠、方針、履歴、候補のどれとして見せるのか。表示形式が agent の使い方を変える。

Rejection

古い記憶、矛盾する記憶、prompt injection 的な記憶をどう無効化するか。保存より拒否が難しい。

Skill は自然言語のままだと腐る

SKILL.md のような自然言語 skill は、人間にも LLM にも読みやすい。一方で、呼び出し条件、実行手順、tool や resource への副作用、必要権限、失敗時の扱いが一つの文章に混ざりやすい。

skill が少ないうちは問題になりにくい。しかし registry が大きくなると、discovery、risk review、重複統合、古い手順の検出が難しくなる。自然言語 skill は便利な作業単位だが、運用が始まると metadata が必要になる。

三層で見る補助線

  • Scheduling layer: いつ呼ぶべきか、どんな入力や状況で有効か。
  • Structural layer: どんな場面・手順・分岐で動くか。
  • Logical layer: どんな atomic action と resource use を持つか。

これは自然言語 skill を捨てる話ではない。読める本文を残しつつ、機械可読な index、risk surface、lifecycle metadata を重ねる方向として使える。

Rot は削除問題ではなく運用問題

skill rot は「古い skill を消せばよい」という話ではない。使用回数、最終使用時刻、patch 履歴、重複、drift、pin、archive、restore を持つ lifecycle 管理が必要になる。

memory も同じで、保存された記憶が多いほど賢くなるわけではない。使われない記憶、今の方針と矛盾する記憶、古い外部情報、意図せず行動を縛る記述は、長期運用では負債になる。

Stale

当時は正しかったが、今は条件が変わった知識。更新日と適用条件がないと混入しやすい。

Duplicate

似た memory や skill が複数あり、どれを優先するか分からない状態。routing が不安定になる。

Unsafe

権限、秘密情報、外部操作、古い依存に関わる手順。呼び出し前の risk label が必要になる。

実務への引き出し

  • catalog は単なるファイル一覧ではなく、memory interface として見る。summary、status、updated、related が retrieval policy の手がかりになる。
  • skill は本文だけでなく、trigger、side effects、必要権限、古さ、使用頻度を metadata として持つ余地がある。
  • wiki lint は整合性確認だけでなく、memory rot を防ぐ curator として育てられる。
  • learn-from-logs のように memory を増やす仕組みは、memory を減らす・統合する仕組みと対にした方がよい。

次の設計問い

Routing

skill の description だけで routing する方式に、どこまで risk label と lifecycle state を足すべきか。

Status

active、draft、unprocessed に加えて、stale や archived を導入するべきか。

Authorship

agent-created skill と human-authored skill を明示的に分け、信頼度やレビュー条件を変えるべきか。

読後に残す一言はこれ。agent の長期運用では、「覚える技術」と同じくらい「忘れる技術」と「腐りを見つける技術」が重要になる。
