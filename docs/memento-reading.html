どんな論文か

Memento: Fine-tuning LLM Agents without Fine-tuning LLMs は、LLM エージェントの「学習」を、モデル重みの更新ではなく、経験の保存・選択・再利用として設計する論文。 狙いは、長い調査やツール利用を行う agent が、毎回ゼロから考えるのではなく、過去の似た課題で何を計画し、何が成功し、何が失敗したかを使って次の行動を良くすることにある。

従来の agent 改善には大きく二つの寄り道がある。ひとつは、固定された reflection や手書き workflow を増やす方向。これは狭いタスクでは効くが、環境が変わると硬い。 もうひとつは、LLM 自体を強化学習や教師あり学習で fine-tuning する方向。こちらは柔軟だが、計算コストが高く、継続的に運用しながら学ぶには重い。

Memento はその間に入る。モデル本体は固定したまま、過去の trajectory を「case」として外部記憶に保存し、現在のタスクに似た case を取り出して planner に渡す。 つまり、学習するのは「どの case を参照すると良いか」という選択方針であり、LLM の重みではない。

この論文が面白いのは、単なる RAG ではなく、古典的な case-based reasoning と強化学習をつないでいるところ。 過去の文書を検索するのではなく、過去の状態・計画・報酬を持つ経験を検索し、それを planning の材料にする。

読む前の見取り図としては、Direct Corpus Interaction が「資料庫をどう歩くか」、Shepherd が「実行履歴をどう分岐・再生するか」なら、Memento は「成功・失敗した実行経験を、次の計画にどう効かせるか」を扱う論文だと思うと入りやすい。

課題と貢献

継続学習を外部化する

LLM の重みを更新せず、経験を case bank に保存して、類似課題の planning に使う。

M-MDP として定式化

通常の MDP に memory space を加え、状態、行動、報酬を持つ case を読み書きする agent として表す。

Deep research agent で実証

GAIA、DeepResearcher、SimpleQA、HLE で、planning、tool use、web research への効果を測る。

主張の芯は「agent を賢くするには、毎回モデルを訓練し直す必要はない。経験をうまく覚え、うまく選ぶだけでも、かなり学習に近い振る舞いが出る」という点。

手法のしくみ

1. 過去の経験を case として保存する

Memento は、現在のタスク、作った計画、結果の成否を case として保存する。論文中では case を状態、行動、報酬の組として扱う。 これにより、単なる会話ログではなく「この状況で、この計画を使ったら、うまくいった / 失敗した」という再利用可能な経験になる。

2. planner が case を読んで計画を作る

新しいタスクが来ると、planner は case memory から関連する過去ケースを読む。非パラメトリック版では意味的類似度で近い case を取る。 パラメトリック版では、どの case が成功に効きやすいかを Q 関数で学び、その値に基づいて case を選ぶ。

3. executor が MCP 経由でツールを使う

planner が作った subtask を executor が実行する。検索、web crawl、画像、音声、PDF、表計算、コード実行などを MCP 経由で扱う構成になっている。 実行中の subtask や tool call は別の memory に保存され、planner は途中結果を見ながら再計画する。

4. 成功・失敗を次の memory 更新に使う

タスクが終わると、結果を case bank に追加する。パラメトリック版では同時に Q 関数を更新し、「どの case が次の計画で役に立つか」の選択方針を改善する。 ここが、単に事例を増やすだけの RAG と違うところ。

検証結果

Memento は GAIA validation で 87.88% Pass@3 を出し、論文中の時点でトップ水準として報告されている。private test でも 79.40%。 DeepResearcher 系の7つの open-domain QA では平均 66.6% F1、80.4% PM で、training-based な DeepResearcher を上回る。

GAIA

長いツール利用と planning の評価。validation で高い値を出す一方、Level 3 のような長い課題はまだ難しい。

DeepResearcher

web research と multi-hop QA で強い。case-based memory が特に効く領域として読める。

OOD generalization

Musique、Bamboogle、PopQA で case-based memory が 4.7 から 9.6 ポイントの絶対改善を足している。

切り分け実験では、online executor だけを足してもノイズが増える場合がある。planning を入れると大きく改善し、さらに case-based reasoning を足すと追加で伸びる。 つまり「外部情報がある」だけでは足りず、「計画」と「経験の選択」が必要だ、という結果になっている。

限界と読みどころ

  • case bank が増え続けると、検索コストや質の劣化が問題になる。論文も swamping problem に触れている。
  • 報酬が正しく付く前提が強い。実運用では「成功したように見えるが危険」な case をどう扱うかが難しい。
  • Deep research agent としての評価が中心なので、会話 memory や個人 assistant memory へそのまま移せるとは限らない。
  • 結果は強いが、GPT-4.1 / o3 / o4-mini と豊富な tool stack を使っているため、memory だけの効果として読むと過大評価しやすい。

読みどころは、memory を「取り出す知識」ではなく「次の計画を選ぶための経験」として扱う点。 Memory And Skill Rot の文脈では、case を増やすだけでなく、どの case を残すか、どの case を危険として退けるかまで考えたくなる。

読後Q&A

Q. Memento は何を fine-tuning しているの？

A. LLM 本体ではなく、過去 case をどう選ぶかの記憶側の方針を更新する。だから「LLM を fine-tuning しない fine-tuning」というタイトルになっている。

Q. RAG と何が違う？

A. RAG は文書を検索することが多い。Memento は、過去の状態、計画、結果を持つ経験を検索し、次の planning に使う。

Q. case-based reasoning って何？

A. 似た過去事例を思い出し、その解き方を今の問題に合わせて使う考え方。Memento はこれを LLM agent の planning に載せる。

Q. M-MDP のポイントは？

A. 通常の状態と行動に加えて、過去経験の memory space を明示すること。agent の行動が現在状態だけでなく memory にも依存する。

Q. parametric memory は LLM の重み更新？

A. 違う。更新するのは case 選択用の Q 関数で、LLM 本体の重みではない。ここを混ぜると読み違える。

Q. どこで一番効いている？

A. 長い planning、web search、tool use、multi-hop QA のように、過去の進め方が次の計画に効くタスクで強い。

Q. なぜ online executor だけでは不十分？

A. 外部情報を取れるだけだと、ノイズや寄り道も増える。planner と case memory が、どの情報をどう使うかを整える。

Q. 失敗 case も保存する意味は？

A. 同じ失敗を避ける材料になる。成功例だけだと、危ない道や無駄な探索を学べない。

Q. Memory And Skill Rot とどうつながる？

A. case bank は増えるほど良いとは限らない。古い case、危険な case、重複 case をどう管理するかが次の問題になる。

Q. Shepherd と比べると？

A. Shepherd は実行履歴を分岐・再生できる基盤。Memento は、保存された経験を次の計画でどう選ぶかに寄っている。

Q. 実務に持ち込むなら？

A. 「過去の成功手順集」を作るだけでなく、どの状況でどの手順を参照すべきかまで学習・評価する設計にする。

Q. 読後に残る一言は？

A. agent の経験はログではなく、次の planning を変える case として保存して初めて効いてくる。

次に読むなら

Shepherd

経験を case として使う前に、実行履歴を分岐・再生可能な基盤としてどう持つかを見る。

Memory And Skill Rot

case や skill が増えた後、古さ・重複・危険をどう管理するかへ進む。

Remember the Decision, Not the Description

memory を「説明の保存」ではなく「意思決定に効く区別の保存」として読む次の候補。
