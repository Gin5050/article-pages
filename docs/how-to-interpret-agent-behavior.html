How to Interpret Agent Behavior Paper Summary

長時間動く agent の軌跡は、成功率だけでは「どう動いたか」「なぜ失敗したか」が見えない。この論文は、ACT*ONOMY という行動分類体系と自動分析 pipeline で、messy な trajectory を共有可能な behavior profile に変える。

この論文の中心問いは、長時間動く AI agent の行動をどう解釈するかである。Codex、Claude Code、web agent、software engineering agent のような system は、計画、推論、検索、ツール利用、失敗、回復、提出までを長い trajectory として残す。しかし最終成功率だけでは、その agent が何をして、どこで迷い、どこで回復し、なぜ失敗したのかは見えにくい。

著者らはこの問題に対して、ACT*ONOMY という agent behavior の taxonomy を提案する。これは 10 actions、46 sub-actions、120 leaf categories からなる三階層の分類体系で、agent の observable runtime behavior を共通語彙として記述するためのもの。論文では、2024年から2026年の agent 論文35本から抽出した565件の behavior description をもとに、grounded theory approach で分類体系を作っている。

ACT*ONOMY は taxonomy だけではなく、trajectory に適用するための自動分析 pipeline も含む。raw trace を observation / thought / action の turn に分解し、behavior-indicating span を抽出し、action--subaction--leaf の label を付け、統計と session summary に集約する。最終的には、agent ごとの behavior profile や、単一 agent 内の失敗パターンを見られるようにする。

実験では、AG2、HyperAgent、SWE-Agent の public trajectory から行動 profile を比較し、それぞれの architecture や task domain に応じた違いを示す。さらに SWE-Agent の成功 trace と失敗 trace を比較し、失敗 run で「検証が必要だと認識し、検証できないことも認識し、それでも submit する」という細かな failure pattern を leaf-level label で可視化している。

読後に残るポイントは、agent observability には outcome metric だけでなく behavioral interpretability が必要だということ。ACT*ONOMY は、mechanistic interpretability の代わりではなく、trajectory の外側から agent の振る舞いを記述・比較・監査するための補完的な言語として位置づけられる。

どんな論文か

How to Interpret Agent Behavior は、AI agent の実行軌跡を人間が読める行動語彙へ変換するための論文である。対象は、agent が runtime に取る observable actions、つまり計画、推論、検索、ツール呼び出し、評価、終了判断、反省、記憶操作などの外から見える振る舞いである。

論文の問題意識は、agent が長く複雑に動くほど、task success rate や pass@1 だけでは設計改善に必要な情報が足りなくなることにある。成功したかどうかは分かっても、効果的な計画だったのか、ループに入ったのか、検証を省略したのか、失敗から回復したのかは分からない。

そこで著者らは、ACT*ONOMY という分類体系と、自動 trajectory analysis pipeline を提案する。これは、agent の raw trace を quote-grounded な behavior labels に変換し、behavior profile として比較できるようにする試みである。

課題と貢献

第一の貢献は、observable agent behavior を記述する三階層 taxonomy である。ACT*ONOMY は 10 actions、46 sub-actions、120 leaf categories からなり、planning、reasoning、retrieval、grounding、executing、evaluating、deciding、reflecting、learning、memory といった行動を分けて扱う。

第二の貢献は、taxonomy の構築方法である。著者らは35本の peer-reviewed agent papers から565件の behavior description を抽出し、Cognitive Architectures for Language Agents の action-space definition を理論的な足場にしつつ、grounded theory approach で分類を作っている。

第三の貢献は、ACT*ONOMY を trajectory に適用する自動分析 pipeline である。trajectory を per-turn の observation / thought / action に分解し、behavior span を抽出し、action--subaction--leaf label を付け、統計・session segmentation・自然言語 summary に集約する。

第四の貢献は、ACT*ONOMY を living taxonomy として公開 repository で管理し、新しい agent behavior に応じて leaf category を追加できる extension protocol を用意している点である。

手法のしくみ

taxonomy 構築は大きく二段階で進む。Phase 1 では、CoALA などの既存理論から seed codebook を作り、behavior description を verb + noun の形に正規化し、LLM-powered discovery judge と6人の共著者の review によって codebook を拡張・整理する。

この過程で、664 candidate sentences のうち off-topic な99件を除き、565 behavior descriptions から V3 を作る。その後、複数 round の共著者 review を通じて V4.1、さらに120 leaf-level instances を含む V4.2 へ発展させる。

Phase 2 では taxonomy の validation を行う。held-out behavior sentences に対する2人の coder の一致は action level で Cohen's kappa 0.87、sub-action level で 0.72。LLM-powered deductive analyst も primary coder と action level 0.74、subaction level 0.71 の一致を示す。

trajectory analysis pipeline では、raw trajectory を observation / thought / action の turn sequence にし、thought と action 内の behavior-indicating spans を抽出する。各 span に action--subaction--leaf label を付け、必要に応じて codebook にない subaction や leaf を提案する。

最後に、annotated trajectory を集計し、session に分割し、何が起きたかを behavior profile として提示する。これにより、人間が長い raw trace を全部読む前に、どの種類の行動が多く、どこに failure precursor があるかを見られるようにする。

検証結果

large-scale analysis では、awesome-language-agents 由来の211本の behavior-related agent papers から抽出された3,455 behavior descriptions に ACT*ONOMY を適用している。高レベル action では Grounding と Reasoning が合わせて40%以上を占める一方、leaf level では long-tail になり、細かな行動語彙はかなり多様である。

across-agent case study では、AG2、HyperAgent、SWE-Agent の public trajectories から各100 action sequence representations、合計300 traces を分析する。AG2 は Evaluating、Grounding、Deciding が相対的に高く、HyperAgent は Reflecting、Reasoning、Memory が高く、SWE-Agent は Executing が大きく高いという profile の違いが出る。

within-agent case study では、SWE-Agent の成功 trace と失敗 trace を比較する。成功 trace は locate the bug、patch the bug、verify the fix、submit の4 phase で、Reasoning と Executing が比較的均衡する。一方、失敗 trace は search、dead end、second dead end、find file、patch/recover/submit の5 phase になり、Reasoning が膨らむ。

特に重要なのは、失敗 trace の leaf-level label が「submit anyway, without verifying」という failure pattern を浮かび上がらせる点である。agent は検証が必要だと認識し、Django server を動かせないため検証できないことも認識し、それでも submit する。この細部は、trajectory 全体の成功/失敗や turn count だけでは見えにくい。

論文は、ACT*ONOMY が across-agent では architecture fingerprint を、within-agent では trajectory-level pattern や failure precursor を可視化できると主張する。

限界と読みどころ

ACT*ONOMY は完成済みの永久 taxonomy ではなく、agent の進化に合わせて育てる starting point として提示されている。top-level actions と多くの sub-actions は比較的安定する想定だが、leaf categories は新しい agent design、tool use、modality に応じて増える。

analysis の品質は raw trace の品質に依存する。thought、tool call、observation、action が十分な粒度で記録されていなければ、behavior span の抽出や label assignment も粗くなる。

taxonomy は研究論文内の behavior descriptions を大きな根拠にしているため、実運用で現れるが論文でまだ語られていない行動は拾いにくい可能性がある。著者らは extension protocol を用意してこの問題に対応しようとしている。

ACT*ONOMY は mechanistic interpretability を置き換えるものではない。内部表現や回路を見るのではなく、trajectory 上の observable behavior を記述する behavioral interpretability であり、将来的には内部状態の分析と接続されるとよい、という位置づけである。

次に読むなら

この論文は、agent の性能を上げる手法というより、agent の振る舞いを読むための語彙と観察面を作る論文として読むとよい。

Counterfactual Trace Auditing と並べると、CTA が skill あり/なしの差分を見るのに対し、ACT*ONOMY はそもそも trace をどう tag 付けして読める形にするかを扱っている、と整理できる。

実務に引くなら、Codex skills や paper-watch の実行ログを、成功/失敗ではなく、planning、retrieval、evaluating、executing、reflecting、memory の分布として眺める mini dashboard を作れないか、という方向に考えるのがよい。

読後Q&A

この論文の中心問いは？

長く複雑に動く AI agent の runtime behavior を、人間が共有できる語彙でどう記述し、比較し、監査するか。

なぜ成功率だけでは足りないの？

成功率はタスクが終わったかどうかを示すが、agent がどう計画し、どこで迷い、どう検証し、どの失敗パターンに入ったかを示さないから。

ACT*ONOMY とは何？

agent の observable behavior を分類する taxonomy。10 actions、46 sub-actions、120 leaf categories の三階層で、trajectory を行動タグとして読めるようにする。

taxonomy はどう作られた？

35本の peer-reviewed agent papers から565件の behavior description を集め、CoALA などの理論を足場にしつつ、grounded theory approach と人手 review で作られた。

10 actions には何が含まれる？

Grounding、Retrieval、Reasoning、Planning、Evaluating、Deciding、Executing、Reflecting、Learning、Memory など、外部との接続、内部思考、実行、適応、記憶操作を分ける。

自動分析 pipeline は何をする？

raw trajectory を observation / thought / action の turn に分け、behavior span を抽出し、action--subaction--leaf label を付け、統計と session summary に集約する。

case study では何が分かった？

AG2、HyperAgent、SWE-Agent はそれぞれ異なる behavior profile を持つ。さらに SWE-Agent 内でも、成功 trace と失敗 trace は phase 構成や Reasoning / Executing の比率が違う。

一番面白い失敗パターンは？

SWE-Agent の失敗 trace で見つかった「submit anyway, without verifying」。agent は検証が必要だと分かっていて、検証できないことも分かっているのに、そのまま提出する。

これは mechanistic interpretability なの？

違う。モデル内部を見るのではなく、trajectory 上の外から観察できる行動を記述する behavioral interpretability である。論文は両者を補完関係として位置づける。

この論文の注意点は？

taxonomy は完成品ではなく starting point であり、leaf category は増え続ける。また、分析の精度は trace の記録粒度と品質に強く依存する。

Codex skills 運用に使うなら？

skill の成功/失敗だけでなく、planning が増えたか、retrieval が偏ったか、evaluating が減ったか、executing が早すぎないか、といった behavior profile の変化を見る。

一言でいうと？

agent の長い軌跡を、読める行動語彙と profile に変える論文。trace auditing や skill rot の議論に、何を観察するかの土台を与える。
